## 1. 前提— numanieの現在の規模（Annict比較のための事実確認）

Annict調査（60,000人・12年運営）と並べて議論する前に、numanie側の実態を確認した。

- **コードベース上、ユーザー数を集計する仕組みが存在しない**（NextAuth は JWT 戦略で `users` マスターテーブルなし。admin/analytics系API・集計スクリプトもゼロ）。
- `MONETIZATION_ROADMAP.md` の KPI目標を見ても **Phase 0（2026年7月）の目標は「DAU: 自分1人でもOK」** であり、numanieは実質**正式リリース前のMVP準備段階**。
- フェーズ別MAU目標（2026-06-09策定、既存）:

| フェーズ | 時期 | MAU目標 |
|---|---|---|
| Phase 0 | 2026年7月 | 1人（自分） |
| Phase 1（ソフトローンチ） | 2026年8-9月 | 100人 |
| Phase 2（コミュニティ形成） | 2026年10-12月 | 500〜1,000人 |
| Phase 3（収益化） | 2027年Q1〜 | 3,000〜5,000人 |

**結論**: Annictの60,000人は12年かけた到達点であり、numanieとは比較できる段階にない。今回の壁打ちは「これからゼロから立ち上げる」前提で行う。

## 3. 決済導線 — Web/ネイティブをどう設計するか

### 3.1 制約の整理
- **アフィリエイトはWeb/ネイティブ問わず問題なし**。外部サービスへの遷移リンクであり、Apple/Googleの「アプリ内デジタルコンテンツ課金」規約（Apple Guideline 3.1.1等）の対象外。現行のIssue #15実装（PR#146）のまま両OSで使える。
- **プレミアム機能（カスタムTierラベル・高解像度PNG・複数シーズン統計等、`MONETIZATION_ROADMAP.md` Phase3-D）の課金は規約制約が強い**:
  - Capacitor（WebView）はアプリ内から直接ストア課金を呼べず、ネイティブ側に購入画面を別途用意する必要がある。
  - Apple/Googleは「アプリ内で消費するデジタル機能」を外部Web決済（Stripe等）に誘導することを原則禁止（= ネイティブアプリ内にWeb決済へのリンクを置くと審査リスク）。

### 3.2 検討した2方向
| 方向 | 内容 | メリット | デメリット |
|------|------|----------|------------|
| **A. Web課金のみ（Netflix方式）** | 課金はWeb上のみで完結。ネイティブアプリには購入ボタンを置かず、ログイン済みユーザーの解除状態を表示するだけ | ストア手数料(15-30%)・審査リスクを回避。実装も軽い | ネイティブ内で新規課金導線に出会えない（コンバージョン機会減） |
| B. 両対応（IAP+Web） | RevenueCat等で両OSのストア課金を実装 | ネイティブ内で購入完結、コンバージョンが良くなりやすい | 手数料・審査対応コストが高い。pre-revenueの現段階では投資対効果が低い |

### 3.3 決定（2026-06-22, オーナー）
**A: Web課金のみを採用**。

- ネイティブアプリ内に課金UIは置かない。プレミアム機能は「Webでログイン→購入→ネイティブアプリでも同じログイン状態なら機能解放」という構成。
- 理由: 収益実績ゼロ・ユーザー基盤もこれから立ち上げる段階で、ストア手数料・審査対応コストを先行投資する理由がない。実需が見えてからBへの移行を検討する。
- アフィリエイト（Issue #15実装済み）は対象外＝そのままWeb/ネイティブ両方で継続。
- `MONETIZATION_ROADMAP.md` Phase 2「IAP」セクションを本決定で更新する（§5参照）。

## 4. ネイティブアプリ特有の差別化機能（アイデアブレスト）

既存計画（プッシュ通知・ウィジェット・Siri・共有シート・ディープリンク — `docs/native-monetization-roadmap-20260620.html` ④⑥、EPIC #79 / #94）に加えて、Annict調査（[market-annict-competitor-20260622.md](../docs/research/market-annict-competitor-20260622.md)）から導いた「Annictにない・カジュアル層にも刺さる」候補。

| アイデア | 効果 | 優先度 |
|---|---|---|
| ホーム画面ウィジェット「今夜の放送」 | 起動不要で確認でき、カジュアル層の習慣化に直結 | 高 |
| 通知から直接「視聴済み」ワンタップ更新 | Annictの手動記録の手間をさらに削減（記録ゼロでも回る、という差別化の核を強化） | 高 |
| 端末標準カレンダーへの放送予定自動登録 | Annictの「外部カレンダー連携」止まりを一歩進める（自動化） | 中 |
| オフラインでもTier表・視聴リスト閲覧 | PWAより確実。通信不安定時の体験向上 | 中 |
| Live Activity / ロック画面での放送中カウントダウン（iOS） | リアタイ派への訴求。視覚的な新規性 | 低〜中（長期） |
| Apple Watch / Wear OS 簡易表示 | 物珍しさはあるが工数対効果が低い | 低 |

優先度「高」の2件（ホームウィジェット・通知ワンタップ更新）は、Expo長期レーンのEPIC #94（プッシュ通知/ウィジェット/Siri・App Shortcuts）の具体スコープとして次回Issue化を検討する余地あり（未起票）。

## 6. 決済プロバイダ検討: Stripe（2026-06-23、issue化）

オーナーから「決済はStripeを入れたい」という方向性が出たため、§3で確定した「A: Web課金のみ」方針を
**具体的にどう実装するか**の壁打ちを開始。検討issue: #160。**本セクション時点では未決定・実装着手前の
論点整理**であり、下記が固まってから実装issueに分割する。

### 論点
1. **課金対象**: `MONETIZATION_ROADMAP.md` Phase3-D（カスタムTierラベル・色テーマ／高解像度PNG出力／
   複数シーズン統計／プロフィール・バッジ）のどれを最初に有料化するか。あるいはC案（Patreon/Ko-fi型
   サポーター、機能制限なしの感謝バッジ）を先に出すか
2. **課金モデル**: 月額サブスク／買い切り／金額自由の投げ銭。Stripe Checkoutの`mode`
   （`subscription` / `payment`）が変わる
3. **技術スコープ**（モデル確定後）:
   - Stripe Checkout Session（redirect）か Embedded Checkout か
   - Customer Portalでのセルフ解約導線
   - DB: §1で指摘済みの通り`users`マスターテーブルが存在せず、NextAuth(JWT戦略)の`token.sub`を
     `userId`として各機能テーブルに直接使う構成。Stripe連携には`userId`→`stripeCustomerId`を
     永続化する新テーブル（例: `user_billing`）が必要
   - Webhook: `checkout.session.completed`/`customer.subscription.updated`/
     `customer.subscription.deleted`/`invoice.payment_failed`を最低限処理。署名検証は
     `request.text()`で生ボディを取得してから`constructEvent()`（`request.json()`では失敗する）。
     イベントID冪等性チェック必須（再送対策）
4. **法的対応**: Web上で有料サービスを販売する場合、特定商取引法に基づく表記ページが必要
   （事業者名・所在地・連絡先・返金条件等）。#87（Privacy Policy/Terms整備）と合わせて整備が必要
5. **税務/Stripe側手続き**: Stripe Japanアカウントの本人確認（KYC）、JPY対応、少額でも個人の
   確定申告対象になりうる点をオーナーが認識しておく

### 優先度に関する注記
2026-06-23決定の#157（Web優先・アフィリエイト/広告の小規模先行導入）により、直近の収益化優先順位は
**アフィリエイトASP審査・広告小規模導入が先**。Stripe決済基盤（Phase3-D系プレミアム機能）は
依然将来フェーズであり、本セクションは実装着手の合図ではなく検討の起点として記録する。
