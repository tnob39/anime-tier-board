### 1.2 Annict 詳細プロファイル

- **運営**: 個人開発（@shimbaco）、12年運営。法人格なし、NPOでもない。「趣味の延長」の継続プロジェクト。
- **収益**: サポーター制サブスク（Stripe）。広告非表示・正確な視聴日時記録が特典。積極的な収益最大化は目指していない。
- **技術**: Rails→Go移行中（chi/sqlc/templ/River）。単一VPS運用、リソース効率重視。Apache License 2.0でOSS。
- **公式ネイティブアプリ: 意図的に作らない**。開発者本人が「APIを充実させてサードパーティクライアントのエコシステムを育てたい」と明言（2020年〜の方針）。非公式アプリ（AniHub等）が複数存在するのみ。
- **機能**: 視聴記録（ステータス・話数・感想）、ソーシャル（フォロー・タイムライン・ネタバレ制御）、発見、放送カレンダー（カレンダー連携）、Web API。
- **機能が無い領域**: Tier/序列づけ機能（×）、サブスク見放題の最適化・診断（△=配信先表示のみ）、自動クロスプラットフォーム同期（手動記録が基本）。

### 1.3 市場全体の構造（Grokの考察）

- 「完璧な一括管理アプリが欲しい」という強い需要シグナルは薄く、**役割特化ツールの併用**が定着（カレンダー=しょぼい／記録=Annict／レビュー=Filmarks/MAL／視聴=配信アプリ）。
- 理由: (1) 役割分担で大半のニーズが満たされてしまう、(2) データソース統合の技術的障壁（放送データは手動/コミュニティ依存、配信APIは非公開）、(3) 個人開発で60kユーザーを維持できる程度の市場規模＝大規模投資のインセンティブが薄い。
- 結果として **「決定版オールインワン」は2026年6月時点で日本市場に存在しない**。

## 2. 分析 — Annict は本当に「最対抗馬」か

### 2.1 強み（侵食しにくい部分）
- **12年・60k人のコミュニティ蓄積**。古参マニア層ほど「過去ログ（視聴記録の積み重ね）」への執着が強く、乗り換えコストが高い。
- **API公開によるエコシステム**: サードパーティ文化・開発者からの信頼。
- **「記録・ソーシャル・発見」の三点はnumanieと同等以上**（PRODUCT_CONCEPT.md §2の表でも視聴記録◎・SNS共有○）。

### 2.2 構造的弱み（Annictが自ら選んでいる制約 = 変わりにくい）
1. **公式ネイティブアプリを作らない方針が思想レベルで固定**（開発者本人が「スマホで見ないから作る気がない」と明言）。これは技術的後れではなく**意図的な選択**なので、Annictが追いついてくる可能性は低い。numanieのネイティブ化ロードマップ（EPIC #79）は構造的に閉じにくい優位。
2. **収益最大化の意思が薄い（サポーター寄付モデル）**。サブスク診断・アフィリエイト導線（Issue #15、実装済み: PR#146）のような「見られるサービスへの誘導で収益化する」レーンに、Annictは正面から参入してこない。
3. **手動記録が基本＝マニア向けの作業コスト**。しょぼいカレンダーの「マークしすぎてバンされた」不満が象徴するように、**コミュニティ手動入力モデルは入力量に比例して摩耗する**。numanieがTMDb/AniList APIベースの自動カレンダーで「入力ゼロでも回る」体験を維持できれば、ここはカジュアル層に対する明確な優位。
4. **Tier/序列づけ機能が存在しない（×）**。「沼の住人」が好きな「自分の中での順位づけ・資産化」という欲求にAnnictは応えていない。これはマニア層に対してもnumanieが取れる差別化。
5. **UIが開発者文化に最適化されている**＝ユーザー(Nobu)の「凝りすぎ」評はここに根拠がある。API・OSS・サードパーティクライアント前提のプロダクト思想は、**カジュアル層には心理的ハードルそのもの**（「自分でクライアント選ぶ/探す」を要求する設計は一般ユーザー向けではない）。

### 2.3 結論
Annictは「マニア層の記録・ソーシャル」では強いが、**カジュアル層を取り込む気がそもそも無い**（ネイティブアプリ拒否・手動記録・開発者文化）。numanieにとっての本当の脅威は「Annictに直接競り負ける」ことではなく、**マニア層がAnnictの蓄積に固定されたまま動かない**こと。カジュアル層に関してはAnnictは実質的に競合していない（Filmarks・配信公式アプリの方が脅威度が高い）。

## 3. 対抗戦略案

### 3.1 カジュアル層 — Annictは無風、本当の競合は配信公式アプリ/Filmarks
- **打ち手は現行方針のままで構造的に有利**: ネイティブアプリ化（EPIC #79）・自動カレンダー（API依存・手動マーク不要）・サブスク診断（Issue #133 N3継続）。Annict対抗という観点では**追加の方向転換は不要**、現行ロードマップを進めるだけでAnnictが踏み込んでいない領域を取れる。
- 訴求コピー候補: 「マークしすぎてバンされない」「記録ゼロでも回る」「アプリひとつでどこで見られるか分かる」— しょぼいカレンダー/Annictの実際の不満ポストを裏取りした言葉なので刺さりやすい。

### 3.2 マニア層 — Annictの蓄積資産にどう向き合うか
マニア層は「Annictをやめてnumanieに乗り換える」より「Annictは記録用に残しつつnumanieも使う」が現実的な移行パス（Grok調査が指摘した「併用文化」とも整合）。
- **(a) 共存前提の差別化**: Tier表（D&D・PNG出力・布教カード）はAnnictに存在しない機能なので、「記録はAnnict、序列化・布教・サブスク最適化はnumanie」という併用ポジションを取る。乗り換えを強要しない分、導入障壁が低い。
- **(b) Annictからの取り込み（要検討・要技術調査）**: Annictは**Web API公開**かつOSS（Apache 2.0）。`developers.annict.com` のAPIを使い、Annictの視聴記録をnumanieにインポートする機能を作れば、「マニア層の積み上げを捨てさせない」形でnumanie側にTier付けさせられる。乗り換えコストを実質ゼロにする一手。
- **(c) 過剰に「凝る」方向では戦わない**: Annictの開発者文化・API/OSSエコシステムを正面から模倣・追随する必要はない。numanieの強みは「マニアでも使える上品な見た目とTier化」であり、「開発者向けAPI公開」のような方向に資源を割くのは現時点では優先度低（ペルソナ「ぬま」が求めているのはTier・分析・布教であり、サードパーティクライアントではない）。

### 3.3 優先度（既存Issue/EPICとの対応）
| 優先度 | 打ち手 | 既存対応 |
|--------|--------|----------|
| 高（進行中を継続） | ネイティブアプリ化 | EPIC #79 / #139, #142 等 |
| 高（進行中を継続） | サブスク診断・分析統合 | Issue #133 N3b以降 |
| 中（新規検討） | Annict APIインポート機能 | 新規Issue化を推奨（要API調査タスク） |
| 低（保留） | API公開・サードパーティエコシステム化 | 現時点では非推奨。マニア層の実需と乖離 |
