> **対象**: anime-tier-board（numanie）のネイティブアプリ（apps/native / anime-tier-board-native）

## 1. 基本方針 — この問題の本質

### 1-1. これは何の問題か

| レイヤ | 該当度 | 根拠 |
|--------|--------|------|
| UIデザイン（見た目） | 低 | 見た目を整えても「どこに何があるか」は解決しない |
| **情報設計（IA）** | **高** | 導線3分散・機能埋没は配置ルール不在の症状 [事実: UX_DIRECTION §1] |
| **機能優先順位** | **高** | 「全部大事」のまま並べるから過多になる |
| ユーザー導線 | 高 | 「次に何をすべきか」がホームで提示されていない |
| プロダクト戦略 | 中 | ブランド（視聴管理）とメインURL（Tier作成ツール）の不一致 [事実: UX_DIRECTION §1-1] |

**結論**: これは UI デザインの問題ではなく、**「機能一覧 → 画面」の直訳をやめ、「ユーザーの目的 → 頻度 → 配置」に変換する情報設計の問題**。

### 1-2. なぜ機能一覧から画面配置を考えると失敗するか

1. 機能一覧は「開発した順のリスト」であり、重要度も頻度も文脈も持たない
2. 全機能が「1票ずつ」持つため、政治的に全部ホームに並ぶ（→本件の禁止事項）
3. 「事業側が見せたい機能」（例: サブスク/アフィリエイト）が頻度の低さを無視して常時表示される
4. ユーザーは機能名で考えない。「今夜何を見るか」で考える

### 1-3. 考える順番（実務用の判断基準）

```
① JTBD定義（ユーザーは何を達成しに来るか）
② 頻度×重要度で機能を再分類（§3）
③ メイン体験を1つ決める（=ホーム）
④ 現行の順序付き **5タブ**を頻度上位の「目的」に割当て（4タブ案は候補にしない）
⑤ 低頻度・高重要は文脈表示/通知/検索/格納へ逃がす
⑥ プロトタイプ→5人テスト→計測→調整
```

**判断基準（迷ったらこれ）**: 「その機能を、週に何回・どんな瞬間に・何のために開くか」を一文で言えないものは常時表示しない。

## 2. ユーザー起点の整理

### 2-1. ペルソナ 4パターン **[仮説]**（利用ログ未提供のため。§9のBigQuery分析で実在比率を検証）

| | P1 今期完走型 | P2 サブスク見直し型 | P3 Tier職人型 | P4 ライト回帰型 |
|---|---|---|---|---|

### 3-3. Impact × Frequency マトリクス

```
                高頻度
        │ カレンダー ◎ホームFV
        │ ステータス更新 ◎常時
  高    │────────────────────
  Impact│ サブスク診断 → 文脈+マイページ経由分析
        │ Tier作成/共有 → タブ+期末文脈
        │────────────────────
  低    │ 声優/更新情報/共有閲覧
        │ → 検索・格納・リンク到達
                低頻度
```

- **高頻度×高Impact** → ホーム/常時
- **低頻度×高Impact** → 文脈表示・通知（※常時表示にしないのが本件の肝）
- **高頻度×低Impact** → 軽量UI（ボトムシート等）で摩擦最小化
- **低頻度×低Impact** → 検索/格納/廃止候補

## 4. ナビゲーション設計方針

> **現行ボトムナビ（唯一）**: ホーム `/` · Tier `/tier` · さがす `/explore` · マイリスト `/watchlist` · マイページ `/mypage`（この順）。
> 認証別の表示・閲覧・保存時ログイン・認証後復帰は [UIUX_RECONSIDERATION_20260711.md](./UIUX_RECONSIDERATION_20260711.md) §5。
> text-first（単独/装飾アイコン禁止・utility は常時可視ラベル併記）は同 §7 および [DESIGN.md](../DESIGN.md)。

| パターン | 採否 | 使い方 |
|----------|------|--------|
| ボトムナビ | ◎ | **5 タブ（上記・正規）**。「週1回以上×目的が一文で言える」機能のみ。ラベル常時表示 |
| ホーム | ◎ | 次アクション提示の場（§5）。機能一覧禁止 |
| セグメンテッドコントロール | ○ | ホーム内「今日/今週」、マイリスト内「視聴中/予定/完了」 |
| 検索ファースト | ○ | 「さがす」タブ上部に固定検索バー（オーナー表示時）。声優・過去作はここから |
| ボトムシート | ◎ | ステータス変更・配信リンク・二次操作。画面遷移を減らす主役 |
| 文脈導線 | ◎ | 期末→Tier作成、月初→サブスク見直し、作品詳細→声優 |
| 通知 | ○ | 放送直前・新シーズン開始・共有への反応。**設定で粒度制御必須** |
| レコメンド | ○ | ホーム「今夜の候補」（空き枠がある時のみ） |
| ハンバーガー/ドロワー | ✗（方針④） | **廃止方向**。設定・アカウント・更新情報は **マイページ**へ集約 [事実: 方針④] |
| もっと見る(ゴミ箱タブ) | ✗ | 不採用。低頻度はマイページへ |
| FAB | ✗ | 主アクション（記録更新）はカード上で完結するため不要 |
| タブバー(上部) | △ | 分析画面内の「サブスク/タイプ」切替のみ（分析自体は Bottom Nav 外） |

**ボトムナビ選定基準**: ①週1回以上使う ②タップの目的が一文で言える ③他タブと目的が重複しない ④50代でも **常時可視ラベル**だけで中身が想像できる（「さがす」「ホーム」等のひらがな/平易語）。アイコンのみは禁止。

## 5. ホーム画面の考え方

**原則**: ホーム=「次に何をすべきか」の提示。機能一覧・開発者メタ情報・即ログイン壁は禁止 [事実: UX_DIRECTION §2]。

### 出す / 出さない

| 出す | 出さない |
|------|----------|
| 今夜〜今週の放送予定（次にやること） | 全機能グリッド |
| 未消化・続きから（未完了タスク) | データソース/キャッシュ等メタ情報 |
| 期末のTier作成プロンプト（文脈） | 常設のサブスク広告枠 |
| 空き時間の「今夜の候補」レコメンド | 分析グラフ（マイページ経由の分析へ） |
| （条件付き）サブスクカバー率の変化 | 設定・管理系 |

### ホームカード設計案（優先順）

| # | カード | 表示条件 | 対応JTBD |
|---|--------|----------|----------|
| 1 | **今夜の放送/配信**（横レーン、配信チップ付き） | 常時（当日分あり時） | P1/P4「今夜何を見る」 |
| 2 | **続きを見る/未消化**（視聴中で遅れあり） | 未消化あり時 | P1 消化漏れ防止 |
| 3 | **今週のカレンダー**（曜日レーン、#67準拠） | 常時 | P1 週の見通し |
| 4 | **今期の注目/追加候補** | 視聴中が少ない時に格上げ | P4/新規 活性化 |
| 5 | **今期Tierを作ろう** | 期末4週間のみ | P3 共有バイラル |
| 6 | **サブスク見直し** | カバー率変化時 or 月初のみ | P2 収益接点 |
| 空状態 | チュートリアル「まず1本、視聴中を選ぶ」 | 未登録時（ゲスト閲覧可・即ログイン壁なし） | 活性化 |

## 6. 情報アーキテクチャ — 現行 5 タブと検討履歴

### 6.0 現行（唯一の採用 IA）

```
[ ホーム / ] [ Tier /tier ] [ さがす /explore ] [ マイリスト /watchlist ] [ マイページ /mypage ]
```

| タブ | ルート | 役割 | 含める | 含めない |
|------|--------|------|--------|----------|
| ホーム | `/` | 今夜/今週の次アクション | カレンダー・続き・状態更新シート | 分析・サブスク常設枠・機能一覧 |
| Tier | `/tier` | 今期ランキング作成 | Tier編集・共有/PNG（二次メニュー内） | 過去作探索 |
| さがす | `/explore` | 発見 | 検索・今期/過去作・生涯ランキング・声優(検索経由) | 設定類。**ナビ表示はオーナー限定** |
| マイリスト | `/watchlist` | 視聴管理直行 | 視聴中/予定/完了・進捗 | 分析グラフ本体 |
| マイページ | `/mypage` | アカウントとその他の受け皿 | 分析入口（→`/dashboard`）・サブスク・声優・設定・更新情報 | Bottom Nav の第6タブ化 |

- **分析 `/dashboard`**: Bottom Nav **外**。マイページのサマリー + 「詳細」から到達。
- **Visual / Simple**: 同一 IA・同一到達機能・同一主要 CTA の表示モード（IA 分岐ではない）。
- **認証別**: [UIUX_RECONSIDERATION_20260711.md](./UIUX_RECONSIDERATION_20260711.md) §5。

### 6.1 配置マップ（5 タブ前提・現行）

- **マイページ格納**: 設定、アカウント、通知設定、更新情報、分析詳細入口、（将来）IAP管理
- **文脈表示**: Tier作成プロンプト(期末)、サブスク見直し(月初/カバー率変化)、声優(作品詳細から)
- **通知/レコメンド**: 放送直前通知、新シーズン開始、共有への反応、今夜の候補
- **検索到達**: 過去作、声優、共有ボード
- **廃止/後回し**: エヴァンジェリストカード(後回し)、声優単独タブ、/lab系(非公開維持)、ハンバーガー常設

### 6.2 （履歴）ボトムナビ 3 案 — 実装参照禁止

> 以下は方針④確定前の検討メモ。**推奨・採用ではない**。案A（4 タブ）・案B・案Cおよび「案A 推奨」はすべて **履歴**。

#### 案A: シンプル運用重視（方針③準拠・**却下/履歴**）

```
[ ホーム ] [ Tier ] [ 分析 ] [ さがす ]     （設定=右上歯車 or ドロワー）
```

- 当時の利点: Web(方針③)と一致しやすかった
- **却下理由**: 方針④で分析をタブ降格、マイリスト昇格、マイページ採用へ

#### 案B: 継続・リテンション重視（**履歴**）

```
[ ホーム ] [ 視聴中 ] [ サブスク ] [ さがす ]    （Tier=さがす内+期末ホーム文脈）
```

- 視聴管理直行の問題意識は現行 **マイリスト** タブで吸収済み

#### 案C: 検索・パーソナライズ重視（**履歴**）

```
[ ホーム ] [ さがす ] [ Tier ] [ マイ ]
```

- 「マイ」集約の問題意識は現行 **マイページ** で吸収済み（#64 時点の不採用は方針④で覆された）

## 7. 主要タスクフロー（5本）

| # | フロー | 目的 | 開始点 | 主要ステップ | 迷いポイント | ショートカット/補助 |
|---|--------|------|--------|--------------|--------------|---------------------|
| 1 | 今夜見るものを決める | 10秒で1本決定 | ホーム | ホーム→今夜レーン→カード→配信リンク | 配信先が分からない | カードに配信チップ常設。放送30分前通知→直接カードへ |
| 2 | 視聴記録を更新 | 消化漏れ防止 | ホーム/通知 | カード長押しorタップ→ボトムシート→ステータス選択 | 「どこで変えるか」不明 | 全カード共通のボトムシート。遷移ゼロ |
| 3 | シーズン頭の視聴計画 | 今期作品を選ぶ | さがす(期初はホームでも誘導) | さがす→今期レーン→詳細→「見る予定」追加 | 数が多く選べない | 注目レーン/ジャンルフィルタ。追加後ホームに即反映 |
| 4 | サブスク診断 | 無駄の確認 | マイページ→分析 or 月初ホームカード | マイページ→分析(`/dashboard`)→サブスク→加入サービス選択→カバー率/独占表示 | 診断の入口を知らない | 月初文脈カード+カバー率変化通知。結果→アフィリエイトリンク |
| 5 | Tier作成→共有 | 今期の総括を共有 | Tier(期末はホーム誘導) | Tier→並べ替え→⋯→共有/PNG→X投稿 | 視聴記録との関係が不明 | 視聴済み作品を自動で未分類プールへ供給 [事実: 既存挙動] |

画面遷移の背骨: `ホーム ⇄ 作品カード(ボトムシート) → 作品詳細 → (配信/声優/Tierへ)`。タブ間の直接依存は作らない。

## 8. 機能過多への対処（配置ルール一覧）

| 分類 | 機能 |
|------|------|
| 常時表示 | 今夜/今週カレンダー、ステータス更新シート |
| タブ内 | Tier編集、マイリスト、検索/探索（さがす）、マイページ |
| ホームカード | 続きを見る、今夜の候補、(文脈)Tierプロンプト、(文脈)サブスク見直し |
| 詳細画面 | 配信リンク一覧、声優、あらすじ、共有ボタン |
| 必要時のみ | PNG出力（二次メニュー）、共有設定、フィルタ類 |

## 11.5 デザイン原則の適用 — apple-design（Emil Kowalski, 2026-07-10 オーナー指示で採用）

> 出典: [事実] github.com/emilkowalski/skills の `apple-design`（WWDC設計/モーション講演由来の17原則）。IA確定後のUI/モーション実装・レビューはこの原則を基準にする。

### 本アプリへの具体適用

| 対象UI | 適用原則 | 具体指針 |
|--------|----------|----------|
| **ステータス変更ボトムシート**（§7 フロー2の主役） | 中断可能性 / 速度ハンドオフ / ラバーバンド | Vaul系実装: ドラッグ1:1追従、離した速度でスプリング継続（damping 0.8 / response 0.3）、閉じかけを掴んだら指に追従。上端はラバーバンド |
| **カードレーン**（ホーム/さがす） | 運動量投射 | フリック時は速度からスナップ先を投射（decelerationRate ≈ 0.998）。scroll-snap + momentum |
| **タブバー/ヘッダー** | 素材と深度 | `backdrop-filter: blur + saturate` の半透明レイヤー、コンテンツは下を通す。境界は1px線でなくスクロールエッジのフェード。**ラベル常時可視（text-first）** |
| **Tierカードのドラッグ** | 直接操作 | 掴んだ位置のオフセットを尊重（中心スナップ禁止）。setPointerCapture + 速度履歴 |
| **カード押下** | 応答性 | pointer-down で即ハイライト（`:active` scale 0.97）。リリース待ちのフィードバック禁止 |
| **メニュー/ポップオーバー** | 空間一貫性 | トリガー元を transform-origin に。入った方向へ出す。逆方向はイージングをミラー |
| **通常のUIトランジション** | スプリング既定値 | damping 1.0（バウンス無し）/ response 0.3–0.4。バウンスは「ユーザーが投げた」文脈のみ |
| **全画面** | reduced motion | `prefers-reduced-motion` でスライド/スプリング→クロスフェード。`prefers-reduced-transparency` で半透明→不透明 |
| **タイポグラフィ** | 光学サイズ | 大見出し: letter-spacing -0.02em / line-height 1.05。本文: system-ui、rem基準でOS文字サイズ追従（50代ユーザー要件とも合致） |

### IA判断にも効く原則（§16 Design foundations より）

- **Purpose**: 「作らないものを決める」= §8 の廃止/後回し判断の根拠
- **Simplicity ≠ minimalism**: 1か所に埋めるのはシンプルではない → 「もっと見る」タブ不採用の根拠
- **具体的ラベル > 汎用ラベル**: ナビは中身が分かる名前に。**常時可視ラベル必須**（text-first）。タブ名の言い換え実験は §9 で行うが、**5 タブ構成・順序・ルート自体は変えない**
- **Wayfinding**: 全画面で「どこにいる/どこへ行ける/どう戻る」に答える。トラップ禁止
- **Process**: インタラクティブプロトタイプで検証（静止画1百万枚より動くデモ1つ）→ §9 の Figma プロト+ /lab ページ検証と接続 [事実: UX_DIRECTION は仮ページ検証を必須化済み]

## 12. 制約遵守チェック

- ✅ 全機能ホーム並べ禁止 → ホームはカード6種+文脈条件のみ
- ✅ ボトムナビ **5 タブ固定**（ホーム/Tier/さがす/マイリスト/マイページ）。旧 4 タブは履歴
- ✅ `/dashboard` は Bottom Nav 外（マイページ経由）
- ✅ 低頻度機能の常時表示なし（サブスクは文脈+マイページ、Tier共有は期末文脈）
- ✅ 事業導線（アフィリエイト）は文脈+通知に限定
- ✅ text-first: アイコン+常時可視ラベル。単独/装飾アイコン禁止
- ✅ Visual / Simple は同一 IA・同一到達機能（表示差のみ）
- ✅ 50代向けラベル: 「ホーム/さがす/マイリスト/マイページ」等の平易語。「Tier」のみ固有名として残置（要ユーザーテスト確認）
- ✅ 統合・非表示・廃止候補を明示（§8）
- ✅ GitHub Issues粒度に落とせる構成
