## 1. 問題意識（なぜ刷新するか）

現状の課題（2026-06-13 オーナー評価）:

1. **アプリのアイデンティティが割れている** — ブランドは numanie /「アニメ視聴を自分のものに」（視聴管理）なのに、メインURL `/` が「今期アニメTier表」（作成ツール）を直接表示している。
2. **導線が3か所に分散** — GlobalNav（上部）/ MobileNav（下部）/ HamburgerMenu（ドロワー）。「探す」「声優」が埋没し、どこに何があるか覚えづらい。
3. **各ページのファーストビューが野暮ったい** — グリッド配置中心で、特にスマホで見栄えが悪い。
4. **ユーザーに不要な情報が出ている** — `12作品 / AniList / キャッシュ` のような開発者向けメタ情報、即ログイン壁など。

## 2. デザイン原則（イケてるアプリ基準）

参照する「イケてる」基準: **Letterboxd / AniList / Crunchyroll / Spotify**。

- **ホームは「自分のもの」が映る場所**。アニメ管理アプリのホームは "Continue Watching"（視聴中の続き）が王道。
- **ファーストビューは横スクロールのカードレーン基調**。グリッドの敷き詰めをやめ、レーンで「見せる」。
- **主要導線は下部バー1本に4〜5個まで**。ハンバーガーは設定・アカウント専用。行き先を入れない。
- **1画面1主アクション**。道具箱（ツールバー）を全部見せない。二次操作は ⋯ に畳む。
- **ユーザーにメタ情報を見せない**（データソース名・キャッシュ状態など）。
- **登録は後**。価値を見せる前にログイン壁を出さない（ゲスト閲覧を許容）。
- **シンプル優先**。「シンプルでいい。イケてる画面構成と導線」がオーナーの要望。

**モーション/デザイン品質基準（2026-07-10 オーナー指示で追加・方針④のABEMAビジュアル基準と併用）**: Emil Kowalski の **apple-design 原則**（WWDC由来の17原則、github.com/emilkowalski/skills）を採用。UI実装・レビュー時の要点:

- **応答は pointer-down で即時**。フィードバックはジェスチャー中も連続（終了時だけアニメしない）。
- **アニメーションは全て中断・逆転可能に**。CSS transition/keyframes ではなくスプリング（Motion等）。既定は damping 1.0（バウンス無し）、フリック等の運動量があるときのみ damping ~0.8。シート/ドロワーは damping 0.8 / response 0.3 目安。
- **ドラッグは1:1追従＋離した指の速度をスプリングへ引き継ぐ**。境界はハードストップでなくラバーバンド。
- **空間一貫性**: 入った方向へ出す。メニュー/シートはトリガー元を transform-origin に。
- **prefers-reduced-motion / reduced-transparency 対応必須**（スライド→クロスフェード等）。
- **タイポ**: 大見出しは負のトラッキング＋詰めた行間、本文は system-ui・rem基準でユーザー文字サイズを尊重。
- 詳細な適用は [docs/native-ia-redesign-20260710.md](./native-ia-redesign-20260710.md) §11.5 を参照。

### 方針② ファーストビュー & ページ別UX刷新 — EPIC #65

FV を**横スクロールのカードレーン基調**に刷新する。

> **⚠️ 検証方針（2026-06-13 オーナー指示）**
> レーン基調の刷新は **やってみないと良し悪しが判断できない**。
> いきなり本番ページを書き換えず、**本番に影響しない仮ページ（プロトタイプ）で先に検証する**。
> - 仮ページ例: `/lab/lane` など本番ナビから外したサンドボックスルート（noindex 済み）。
> - 仮ページで `<CardLane>` 共通コンポーネント（#66）とレイアウト案を試し、OK が出てから本番ページへ適用。
> - 検証用の仮ページ・実験コードは本番導線・サイトマップに載せない。採用後に撤去 or `/lab` 配下に隔離。

## 5. エージェント共通の作業規約（UIUX 変更時）

- 実装前に本書 + 対象 Issue を読む。方針逸脱しそうなら先に本書を更新して合意を取る。
- **②レーン刷新は仮ページ検証が必須**。本番ページの直接書き換えから始めない。
- 変更は機能ごとに worktree / ブランチを分け、`npx tsc --noEmit && npm run build` を通してから完了とする。
- ユーザー向けに意味のある変更は `app/updates/page.tsx` の `RELEASES` に追記し最新版に `isLatest: true`（CLAUDE.md ルール）。
- 開発者向けメタ情報（データソース名・キャッシュ・件数デバッグ等）を UI に出さない。
- `Hermes` ファイルはコミット禁止。

## 7. プリフェッチ調査メモ（2026-06-19・要対応／別Issue化予定）

> 「Tier表のバックグラウンド先読み」「カード押下時の先読み」が効いていない疑い、の調査結果（コード根拠あり）。

**現状の仕組み**
- `lib/seasonal-anime-client-cache.ts`: 今期アニメをモジュールレベル `Map` にキャッシュ（TTL 10分・dedupe あり）。
- `lib/use-seasonal-prefetch.ts`: ホームマウントで先読み。**SSR データ seed はログイン済みホーム（`HomeClient`）のみ**だが、**クライアント先読み（`requestIdleCallback` timeout 3s）はゲストホーム（`HomeGuest`）でも実行**される（`app/home-guest.tsx` も `useSeasonalPrefetch` を呼ぶ）。
- `components/TierBoardApp.tsx`: `fetchSeasonalAnimeClient(seasonYear, season)` でキャッシュ優先取得。既定 season は `getCurrentAnimeSeason()`（ホーム先読みのキーと一致）。

**効いていない原因（仮説）**
1. **キャッシュはページロード単位のメモリ**。`/tier` 直アクセス／リロード／フルナビ後はキャッシュ空 → 毎回フル取得。先読みが効くのは「ホーム→Tier をクライアント遷移（リロード無し・10分以内）」した時のみ。
2. **`app/tier/page.tsx` は SSR で種を蒔いていない**（`<TierBoardApp/>` のみ）。ホーム未経由の Tier 直アクセスでは先読みの恩恵ゼロ。
3. **`components/GlobalNav.tsx` のロゴが `<a href="/">`**（フルリロード）→ モジュール `Map` が毎回消える。
4. **カード押下に先読みが無い**。ホームのカードは `<button onClick={() => router.push("/watchlist")}>`（`<Link>` でない）→ Next.js の自動プリフェッチも `router.prefetch`/データ先読みも無し＝「押した時のバックグラウンド呼び込み」は**未実装**。
5. Tier はキャッシュヒット時も `readRemoteBoard`（ネットワーク）を別途叩くため、体感が改善しにくい。

**推奨対応**
- `/tier` を SSR で seed（ホーム同様 initial データ注入）or Tier 遷移時にデータ先読み。
- ホームのカードを `<Link>` 化、または pointerenter で `router.prefetch` + データ先読み。
- 季節キャッシュを `sessionStorage` に永続化し、リロード/直アクセスでも TTL 内なら再利用。
- `GlobalNav` ロゴを `<Link>` 化しフルリロードを止める。
- ホーム先読みと Tier 既定の season キー一致を回帰テストで担保。
