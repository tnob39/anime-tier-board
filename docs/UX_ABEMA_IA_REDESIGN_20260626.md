## 1. なぜ転換するか（方針③ → 方針④）

方針③は「単一IA・4タブ（ホーム/Tier/分析/さがす）／`/watchlist` はタブにしない／マイページ不採用」だった。
2026-06-26、オーナー評価で **ABEMA を参照デザイン言語に採用し、IA 全体を Abema 型へ再設計する** ことを決定。

**ねらい**:

1. 視聴管理リストを「ただのリスト」から「美しいアニメアプリらしい体験」へ進化させ、継続利用率とデータ入力意欲を高める。
2. プロ級アニメ配信アプリ（ABEMA）のメンタルモデル＝**大型ポスターカード + 一目でわかるステータスバッジ + セクション分け + 5タブのボトムナビ + マイページ**を踏襲し、「アプリ感」を一段引き上げる。
3. 将来の Capacitor ネイティブ化（`reference_native_capacitor_repo` / EPIC #79・#139）を見据え、モバイルでアプリライクな体験を最優先にする。

> **方針③からの主な変更点（要確認の影響）**
> - **`/watchlist`（マイリスト）をボトムタブに昇格**（方針③では「タブにしない」だった）。
> - **マイページを採用**（方針③では「不採用」だった）。
> - **分析（`/dashboard`）をボトムタブから降格**し、マイページ配下のサマリー + 詳細導線に再編。
> - ボトムナビを **4タブ → 5タブ** に拡張。

## 2. デザイン原則（ABEMA 基準）

参照する「イケてる」基準: **ABEMA**（補助的に Letterboxd / AniList / Crunchyroll / Spotify）。

- **完全ダークモード**。`#0F0F0F` 背景 / `#1A1A1A` カード / 白・グレー文字。アクセントは控えめなシアン or バイオレット系（既存トークンと整合させる。新規ハードコード色は避け CSS 変数化）。
- **大型ポスターカード中心**。画像 + 下部グラデーションオーバーレイ + ステータスバッジ。角丸 12–16px、微シャドウ。
- **セクション分け / 横スクロールレーン基調**（グリッド敷き詰めをやめる）。方針③の「レーン基調」「`<CardLane>` 共通部品（#66）」を継承。
- **主要導線はボトムバー 5タブ**。それ以外（設定・声優・更新情報・サブスク詳細）はマイページ配下に集約。
- **1画面1主アクション**。二次操作は ⋯ メニューに畳む（既存 watchlist の `MoreVertical` メニューを踏襲）。
- **ユーザーにメタ情報を見せない**（データソース名・キャッシュ・件数デバッグ等）。※方針③原則を継承。
- **モバイルファースト**。PWA→ネイティブ移行でそのまま活きる構造に。
- **既存機能を失わない**。現 `/watchlist` の ★お気に入り度・いつ見る（watchSlot）・メモ・配信サービスチップ・共有/布教カード・放映カレンダーは再設計後も保持する。

## 3. 新 IA / ルーティング（方針④の中核）

### 3.1 ボトムナビ（5タブ）

`components/MobileNav.tsx` を 4タブ → **5タブ**へ。

| 順 | ラベル | ルート | アイコン(案) | 役割 |
|----|--------|--------|--------------|------|
| 1 | ホーム | `/` | `Home` | 放映カレンダー / Continue Watching |
| 2 | Tier | `/tier` | `Table2` | Tier 表 |
| 3 | さがす | `/explore` | `Search` | 探索・ランキング |
| 4 | マイリスト | `/watchlist` | `ListChecks` 等 | 視聴管理ハブ（**昇格**） |
| 5 | マイページ | `/mypage`(新規) | `User`(アバター) | アカウント / データサマリー / 設定入口 |

> **検証必須**: スマホ幅で 5タブはラベルが詰まりやすい。アイコン+短ラベルで実機確認し、収まらなければラベル短縮（例「マイ」）かアイコンのみ+選択中ラベルを検討。

### 3.2 降格・集約するページの行き先

| ページ | 現在 | 方針④での扱い |
|--------|------|----------------|
| 分析 `/dashboard` | ボトムタブ | **マイページ内「視聴データサマリー」**（総視聴・完了数・Tier分布等）。詳細は `/dashboard` へ遷移。`?section=subscriptions` のサブスク分析もマイページ経由に。 |
| サブスク | ハンバーガー(`/dashboard?section=subscriptions`) | マイページ配下（データサマリー/サブスク管理）。 |
| 声優 `/voice-actors` | ハンバーガー | マイページ配下のリスト行。 |
| 更新情報 `/updates` | ハンバーガー | マイページ配下のリスト行。 |
| 設定 `/settings` | ハンバーガー | マイページ配下のリスト行（アカウント管理セクション）。 |

### 3.3 ハンバーガーメニューの扱い

- **原則廃止**し、`components/HamburgerMenu.tsx` の項目（サブスク/声優/設定/更新情報）を**マイページに統合**する。
- 5タブのマイページが「その他導線の受け皿」になるため、ドロワーは不要になる見込み。
- ただし PC 幅のヘッダー導線（`GlobalNav`）との整合は別途確認（PC ではマイページ相当をヘッダーのアバターメニューに出すか要検討 → §7 未決事項）。

### 3.5 現状維持＋切替（フィーチャーフラグ方針）

大改修だが**現状UIを残したまま新UIへ切替可能**にする（オーナー要望 2026-06-26）。

- **既存 `WatchlistClient` は無変更**（フォールバック）。新規 `WatchlistClientV2` を別ファイルで作る。
- 切替は **設定ページの可視トグル**（`app/settings/settings-client.tsx` に「新しいマイリストを試す」）。状態は **`localStorage`** に保持（端末ごと）。
- 共有フラグヘルパー `lib/watchlist-flag.ts`（localStorage キー read/write + `useWatchlistV2()` フック）を新設し、設定トグルと表示切替で共用。
- `app/watchlist/page.tsx`（server）は `initialItems` を取得し、クライアントの切替ラッパー `WatchlistSwitch` に渡す。ラッパーがフラグを読んで V1/V2 を選択（初回SSRは V1 既定→マウント後にフラグ反映。ちらつきは許容、必要なら後で inline script 化）。
- **方針③で撤去した simple/pro モードとは別物**。これは恒久モードではなく**一時的なロールアウトフラグ**で、新UI定着後に削除する。
- ナビ5タブ・マイページ（A2/A3）は影響範囲が広いので **別フラグ**で段階導入（watchlist 単体の S1 とは分離）。

### 3.4 ルーティング変更の波及（必ず追従）

- `components/MobileNav.tsx`（4→5タブ、active 判定、`/` の exact 特例）
- `components/HamburgerMenu.tsx`（廃止 or 縮退）
- `components/GlobalNav.tsx`（右上アイコン/ロゴ遷移先・PC 用マイページ導線）
- `app/mypage/page.tsx`（**新規**）
- `app/dashboard/*`（マイページからの遷移・サブスクセクション導線）
- **PWA App Shortcuts（manifest）** が指すルート
- 共有URL（`/share/...`, `/watchlist/share/...`）・localStorage の storageKey がパス前提で破綻しないか
- `lib/ui-mode.tsx` は方針③で撤去済み（モード分岐なし）。本変更でも単一IAを維持。

## 4. `/watchlist`（マイリスト）再設計詳細

> 現状（`app/watchlist/watchlist-client.tsx`）は「放映カレンダー + カードグリッド」。カードは ★お気に入り・配信チップ・watchSlot・メモ・共有/布教カードを既に保持。**これらを失わずに**ビジュアルとセクション構造を Abema 型へ刷新する。

### 4.1 アニメカード刷新（最重要 / Critical）

- 各作品を**大型ポスターカード**化。画像上部に Tier バッジ or 個人ランク、右上にステータス（視聴中/完了/未視聴）。下部にタイトル + 話数進捗。
- 下部グラデーションオーバーレイ + ステータスバッジ。角丸 12–16px、微シャドウ。
- ホバー/タップでスケール + クイックアクション表示（既存 ⋯ メニュー＝布教カード/視聴解除を踏襲）。
- ABEMA のランキングカード/ポスターカード表現を踏襲。
- 共通部品化を意識（方針③ `<CardLane>` #66 と整合。可能なら `<AnimeCard>` を抽出）。

### 4.2 レイアウト & セクション分け（High）

1本の長いリストではなく以下に分割（タブ or 横スクロールレーン）:

- **現在視聴中**（続きが見やすい優先度高 / Continue Watching）
- **最近追加 / 気になっている**（planned）
- **視聴完了**（completed）
- **保留中**（paused / dropped）

各セクションに「もっと見る」 or 横スクロール。ABEMA の「週間ランキング」「テーマ別ラインナップ」のセクション表現を参考。

### 4.3 ステータスバッジ & 進捗（High / 一部は新データ必要）

- 一目でわかるバッジ: 「視聴中」「完了」「未視聴話あり」「Tier S/A/B」。
- 話数進捗: 円形プログレス or バー + 「12/24話」。
- 「最新話まで視聴済み」等の達成バッジ。
- ABEMA の「全話無料」「最新話」「新着」バッジの視認性・配置を参考。
- **注意**: 「12/24話」「最新話まで視聴済み」は現データモデルに無い（§6 データモデル拡張が前提）。データが揃うまでは放映情報（`airing.nextEpisode` / クール）ベースの簡易表示に留める。

### 4.4 ヘッダー & フィルタ（Medium）

- ヘッダーを sticky に。タイトル「マイリスト」+ アバター + 検索バー。
- フィルターチップを横スクロール（すべて / 視聴中 / 予定 / 完了 / 未視聴話あり）。
- ソート: 追加日 / 最終視聴日 / 進捗 / タイトル。

### 4.5 発見導線の統合（Medium）

- マイリスト上部 or 下部に「あなたの好みに合いそうな新着」「同じジャンルのおすすめ」横スクロール行（`/explore` へ誘導）。

### 4.6 Empty State（Medium）

- イラスト + 「まだ作品がありません」+「作品を探す」ボタン + 人気ランキングから追加リンク。
- 現状 `watchlist-empty` セクションを Abema 風に強化。

## 5. マイページ（`/mypage`）設計

ABEMA のマイページを参照。**アイコン左揃えのクリーンなリスト行 + 右矢印**。カード風の囲みでグループ化。

提案構成:

1. **アカウント管理** — ID 表示 / メール連携 / ログアウト（現 `/settings` 相当）
2. **視聴データサマリー** — 総視聴時間 / 完了作品数 / Tier 分布グラフ（現 `/dashboard` のアニメ分析サマリー）。「詳細を見る」で `/dashboard`。
3. **サブスク管理** — 加入サービス / カバー率（現 `/dashboard?section=subscriptions`）。
4. **マイリスト管理** — `/watchlist` への導線・件数。
5. **視聴履歴** — （新規 / `last_watched_at` 前提。データ揃うまで保留可）
6. **声優** — `/voice-actors`。
7. **エクスポート/インポート** — AniList 連携（将来）。
8. **更新情報** — `/updates`。
9. **設定** — 通知・表示など。

## 6. バックエンド使い回し（重要：再設計はほぼ純フロント）

**2026-06-26 コード確認の結論: バックエンドはほぼ100%再利用でき、スキーマ変更（マイグレーション）は不要。**

現 `AnimeStatusRecord`（`lib/statuses.ts`）は再設計に必要なフィールドを既に持つ:

| 再設計で使うもの | データ source | 状態 |
|------------------|---------------|------|
| セクション分け / ステータスバッジ | `status`（planned/watching/completed/paused/dropped） | ✅ 既存 |
| **進捗バー「6/12話」** | **`watchedEpisodes`** ÷ `anime.episodes`（`lib/types.ts`） | ✅ **既にバックエンド対応済み**（`watched_episodes` カラム・`PUT /api/watchlist` 対応。現UI未露出だが `home-client.tsx` で利用実績あり） |
| ★お気に入り / いつ見る / メモ | `favoriteLevel` / `watchSlot` / `notes` | ✅ 既存 |
| 最終更新ソート | `updatedAt` | ✅ 既存 |
| 共有 | `POST /api/watchlist/shares` | ✅ 既存 |
| おすすめ行 | seasonal/explore API | ✅ 既存 |

**API 契約（変更不要・そのまま使う）**:
- `GET /api/watchlist` → `{ items: AnimeStatusRecord[] }`
- `PUT /api/watchlist` → `{ animeId, favoriteLevel?, watchSlot?, notes?, watchedEpisodes? }`（または `{ animeId, watchRhythm }`）
- `PUT /api/statuses` → `{ animeId, status, anime }`（ステータス変更）
- `DELETE /api/statuses?animeId=...`（視聴解除）
- `POST /api/watchlist/shares` → `{ shareId }`

**唯一の新規読み取り（任意・後回し可）**:

| 項目 | 必要なもの | 優先 |
|------|------------|------|
| Tier バッジ（カード上の S/A/B） | 「作品→Tier」結合（`/api/boards` を読む） | Low（コア再設計には不要・S3で対応） |
| `personalScore` | 既存 `favoriteLevel`（★1-5）で代替 | 不要 |
| 「最終視聴日」厳密値 | 当面 `updatedAt` で代替（厳密化は将来） | Low |

> 結論: **進捗表示を含め B1/B2 はバックエンド無変更で実装可能**。Tier バッジのみ読み取り結合が要るため S3 に分離。

## 7. 未決事項 / 要確認（実装前に解消）

1. **5タブの収まり（モバイル）** — ラベル詰まり。実機検証で短縮/アイコン方針を確定。
2. **PC（GlobalNav）でのマイページ導線** — ヘッダーのアバターメニューに出すか、ハンバーガーを PC のみ残すか。
3. **分析のボトム降格の是非** — 分析を日常的に見るユーザーには遠くなる。マイページ最上部に出すなどで緩和するか。
4. **Tier をボトムに残すか** — Abema 系にTierタブは無いが本アプリのコア機能。残す前提（要再確認）。
5. **ゲスト時のマイリスト/マイページ** — 未ログインでタブを出すか、ログイン誘導にするか（方針③「登録は後」と整合）。
6. **進捗トラッキング（話数）UI** — 加算ボタンの置き場所（カード上 / 詳細）。データモデル拡張とセットで設計。
