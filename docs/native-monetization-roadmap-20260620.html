Anime Tier Board — ネイティブ化 ＆ 収益化 方針

夏アニメ（2026年夏）に間に合わせる「最速でアプリを出す」計画

2026-06-20 ／ Claude + Codex レビュー統合版

① まず一言でいうと

いまある Web アプリを“そのまま”アプリの皮で包んで（=Capacitor）、まず Android から出す。 機能は サブスク診断 と 放映カレンダー に絞り、紹介リンク（アフィリエイト）で収益化する。 そして入口として、Tier表を「人気＋自分の視聴」つきで SNS にシェアし、新規ユーザーを呼び込む（＝成長エンジン）。

2レーン速さのCapacitor / 品質のExpo

🔥 成長エンジン：シェアで広がる（最小構成でも必須）

このアプリの“入口”は、Tier表を SNS にシェアして新しい人が入ってくる流れ。 だから最小構成でも、この拡散ループは外しません。

📊Tier表を作る今期を並べる
→
🔥人気＋自分の視聴が一目で
→
🖼️シェアカード画像で生成
→
📣SNSへX / LINE / IG
→
🙋友達が流入新規ユーザー
💡 朗報：この共有機能はもう全部 Web にあります。 共有ページ（/api/shares）・リアクション/コメント・シェアカード画像（エヴァンジェリストカード＝OG画像）まで実装済み。 Capacitor で包めばそのまま native に乗るので、追加コストはごく小さい。

ネイティブで足すのは 2つだけ：

  • ワンタップ共有（共有シート） — タップ一発で X / LINE / Instagram へ（@capacitor/share）
  • 共有リンクがアプリで開く（ディープリンク） — 友達がリンクを踏むとアプリが起動して回遊（@capacitor/app）

＝ ファネル：入口（獲得）＝Tier＋シェア → 価値（滞在）＝カレンダー → 収益＝サブスク診断＋紹介リンク。

② なぜこれが最速なのか（かみ砕き）

🍱 Capacitor＝お弁当箱に詰めるだけ。 料理（＝今すでに本番で動いている Web アプリ）は完成済み。 それを「アプリの皮（お弁当箱）」に入れてストアに並べるだけなので、中身を作り直さない。 だから Tier・視聴管理・サブスク・カレンダーなど全機能がそのまま入る。
🍳 Expo（今ある別アプリ）＝一から作り直して高級な器に盛る。 通知・ウィジェット・Siri など“本物のネイティブ品質”が出せるが、画面を1枚ずつ作り直すので時間がかかる。 → 捨てずに「長期レーン」として並走させる。

③ 2つのレーンで進める

🚀 高速レーン（Capacitor）

夏に出す。今のWebを包むだけ。

  • 本番Webをそのまま表示
  • Android は Windows で直接ビルド
  • iOS は後追い（クラウドMacビルド）
  • 更新は Web デプロイで即反映（再審査ほぼ不要）
  • シェア／SNS拡散の仕組みも既存のまま乗る

🏆 長期レーン（Expo・既存）

本物のネイティブ品質を後から。

  • 既に5画面＋認証＋通知＋ウィジェット＋Siri実装済
  • プッシュ通知・ウィジェットが強い
  • 急がず品質を上げる枠
  • 夏MVPの担当ではない

※ 旧方針の「Windowsだと iOSが作れない」は、クラウドMacビルド（Codemagic等）で解決できるため、Capacitor採用の障害ではなくなりました。

⑤ 収益化の考え方（かみ砕き）

Phase 1（夏）

紹介リンク（アフィリエイト）が主役

  • 「あなたの作品を一番カバーするサービス」を出し、そのリンクから加入してもらう
  • 押し売りしない：診断の中だけに表示・煽らない
  • 並び順はカバー本数で固定（報酬額で並べ替えない＝信頼維持）
  • コードは小さい。本当の時間は「提携審査」（数日〜数週）
Phase 2（夏の後）

アプリ内課金（IAP）を追加

  • RevenueCat を使って Apple / Google の課金を扱う
  • 決定ゲート：Expo先 か Capacitor先 を1つ選んでから着手
  • Capacitorは WebViewから課金を呼べない → 購入画面はネイティブで用意
  • 権限保存テーブルは user_entitlements（既存の user_subscriptions と名前衝突回避）

⑥ ロードマップ（どう進めるか）

今すぐ
Phase 0

下ごしらえ（並行で着手）

  • 提携申請を今日から：Amazon アソシエイト / A8.net / バリューコマース（審査に時間がかかる＝最優先）
  • 認証スパイク（1〜2日）：Capacitor の中で Google ログインが通るか検証（唯一の技術的な未知）
  • Google Play 登録 $25 ／ Privacy Policy・利用規約・AniList商用ライセンス

ここが詰まると全部止まる“律速”。コードより先に動かす。

1〜2週
Phase 1

夏MVPを作って Android で出す

  • ① 認証移行：/api/subscriptions を Cookie/アプリ両対応に（最初の必須作業）
  • ② 紹介リンクのデータ投入：仕組みはもう実装済（URLを入れるだけ）
  • ③ 診断API：必要な項目だけ返す“公開DTO”を新設（内部情報を漏らさない）
  • ④ 計測リンク：/api/go/affiliate でクリックを数えられるように
  • ⑤ Capacitorで包む → Android内部テスト配信
  • ⑥ 拡散ループをネイティブ強化：ワンタップ共有シート＋共有リンクのディープリンク（共有機能自体はWeb既存）
  • 仕上げ：日本語の文字化け（mojibake）修正

作業は5つの小さなworktreeに分割（バラバラに並行できる形）。

夏の後
Phase 2

iOS 公開 ＋ アプリ内課金

  • iOS：Apple Developer $99 ＋ クラウドMacビルドで TestFlight → 審査
  • IAP：RevenueCat（Expo先/Capacitor先を決定）・user_entitlements・プレミアム機能の定義
継続
長期

Expoレーンで本物ネイティブ品質

  • プッシュ通知（放映・新エピ）・ウィジェット・Siri を磨く
  • Capacitorで出した後、体験を底上げ

⑨ リスクと対策

リスク対策
Capacitor内の Google ログインが弾かれるDay1にスパイク。ダメなら既存のアプリ用ログイン（Bearer）を流用
Appleの「ただのWeb」審査拒否（4.2）通知・共有などネイティブ機能を足す＋Android先行でリスク回避
提携審査が間に合わない今すぐ申請。リンクが空でもアプリは動く作りにして遅延を吸収
配信タイミングを過大表現「放送日」ベースと明記し、解禁時刻は約束しない

⑩ 今日やる “たった2つ”

① アフィリエイト提携を申請する（Amazon / A8.net / バリューコマース）— 一番時間がかかるので最初に。

② Capacitorの認証スパイクを回す（1〜2日）— ここが通れば、あとは定型作業。
