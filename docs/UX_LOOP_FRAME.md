# UX改善ループ 固定分析フレーム

> **位置づけ**: /loop 等で UI/UX 改善を反復するときの**固定の分析フレームと出力契約**。
> ペルソナを毎回ロールプレイし直さない・改善案を乱発しない・Codex に渡せない抽象論を書かない、の3点を強制する。
> 制定: 2026-07-04（オーナーのトークン効率フィードバックに基づく）

## 1. 固定ペルソナ（再定義禁止 — 参照のみ）

ペルソナの正本は **[PRODUCT_CONCEPT.md](./PRODUCT_CONCEPT.md) §3**。分析時は名前で参照するだけにする（説明の再生成禁止）。

| 名前 | 一言 | 分析でチェックする観点 |
|------|------|--------------------|
| **つん**（ライト・シンプルモード） | 今期2〜3本ながら見。サブスク1〜2個 | 今夜/今週の即答性・手数・迷わなさ・専門用語ゼロ |
| **ぬま**（ヘビー・プロモード） | 今期10本超。評価/分析/布教 | 情報密度・一覧性・記録の表現力・共有の映え |
| （共通軸） | **サブスク派が主軸**（確定） | 「配信で今見られる未視聴」への到達コスト |

## 2. 反復あたりの出力契約（厳守）

1回の分析反復で出すのは **High優先の改善提案 最大3件 + 良い点1件** のみ。Medium/Low はタイトル1行の箇条書きでバックログ行きにする（詳細化しない）。

各提案は必ずこのフォーマット:

```markdown
### 提案: <画面名> — <一言>
- 優先度: High（このフレームでは High のみ詳細化）
- 影響範囲: <ファイル/コンポーネント名まで具体的に>
- 想定ユーザ影響: <どのペルソナが・どの頻度で・何が楽になるか 1-2文>
- 実装難易度: Low / Medium / High（変更行数の目安付き）
- 推奨アクション: <Codex Decision Packet に転記できる粒度の具体的変更内容>
```

**禁止**: 「もっと直感的に」「迷わないように」等の抽象表現。UI 変更は「どの要素を・どこへ・どの文言で」まで書く。

## 3. 実装への接続（既存プロセスに乗せる）

1. 提案の最上位1件 → GitHub Issue（症状/やること/対象外/検証）
2. Issue → `docs/CODEX_WORKTREE_GUIDE.md` の **Decision Packet** に変換 → worktree + ローカル Codex
3. 検収（差分逐行・mojibake・CSS変数実在）→ tsc/build は親側 → `/updates` 追記 → マージ → デプロイ確認
4. トークンコストを Issue にコメント（Codex は exec ログ実測、Claude は変更規模を代理指標）

## 4. トークン効率ルール（2026-07-04 実測に基づく）

- **Codex 委任の固定オーバーヘッドは約3万トークン**（コードベース読み込み）。目安:
  - **±20行未満の極小修正 → Claude が直接実装**（委任しない）
  - **+80〜240行の中規模スライス → Codex 委任が最適**（実測 2.5〜4.5万tok）
  - それ以上 → スライス分割
- ペルソナ・設計方針・過去の改善履歴は**このファイルと PRODUCT_CONCEPT / UX_DIRECTION への参照で済ませ、本文へ再展開しない**
- 分析反復と実装反復を分ける場合、分析1回につき実装は最大3スライスまで（発散防止）
